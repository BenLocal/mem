# Agent Instructions

Local-first Rust memory service for multi-agent workflows. Loaded by Claude Code and other agents (`CLAUDE.md` is a symlink to this file).

---

## Common Commands

```bash
cargo run                                      # default = `mem serve` (axum HTTP on 127.0.0.1:3000)
cargo run -- serve                             # explicit HTTP server mode
cargo run -- mcp                               # stdio MCP server; forwards to MEM_BASE_URL

cargo test -q                                  # full suite (integration tests in tests/)
cargo test --test search_api                   # single integration test file
cargo test ingest::compute_content             # single test fn (path or substring)
cargo fmt --check                              # required by CI
cargo clippy --all-targets -- -D warnings      # required by CI

cross build --release                          # cross-compile (reads ./Cross.toml)
```

**Key env vars:** `MEM_DB_PATH` (Lance dataset dir — legacy `mem.duckdb` directory name kept for compat; it holds Lance datasets, not a DuckDB file), `BIND_ADDR` (HTTP bind), `MEM_BASE_URL` / `MEM_TENANT` (MCP forwarder), `MEM_MCP_EXPOSE_EMBEDDINGS=1` (admin tools), `MEM_TRANSCRIPT_EMBED_DISABLED=1` (stop the transcript embedding worker), `MEM_TRANSCRIPT_OVERSAMPLE` (transcript search candidate fan-out factor, default 4; read live in `TranscriptService::search`, invalid values silently fall back to default), `EMBEDDING_BATCH_SIZE` (embedding worker per-tick claim count, **default 8** since 2026-05-21 — flipped from 1 to amortise per-tick read/refresh overhead across more jobs; set to `1` to restore per-job failure isolation), `EMBEDDING_WORKER_POLL_INTERVAL_MS` (embedding worker tick cadence, **default 10_000** since 2026-05-21 — flipped from 1000 after measuring 510% idle CPU + 800+ tokio blocking threads at 1 Hz cadence under the old DuckDB read engine. 10 s tick measured 56% CPU + 217 threads on the same workload. Set to `1000` for sub-second job pickup latency at the cost of higher CPU baseline), `MEM_VACUUM_DISABLED=1` / `MEM_VACUUM_INTERVAL_SECS` / `MEM_VACUUM_OLDER_THAN_DAYS` (Lance manifest pruning, see `src/worker/vacuum_worker.rs`) / `MEM_VACUUM_AGGRESSIVE=1` (opt INTO `delete_unverified=true`; **default OFF** since 2026-06-04 — aggressive prune bypassed Lance's in-flight floor and deleted a manifest the in-flight commit path still referenced, and lance 3.0.1's `conflict_resolver` `.unwrap()`s the `NotFound` → serve core-dumps. `mem serve` has several concurrent writer tasks, so it is NOT single-writer. Legacy `MEM_VACUUM_PRESERVE_UNVERIFIED=1` still forces the safe default and wins over `MEM_VACUUM_AGGRESSIVE`), `MEM_AUTO_PROMOTE_DISABLED=1` (opt OUT of the `PendingConfirmation → Active` sweep — it is **default ON**: worker spawns, promotes `experience`/`implementation`/`episode`/`diary` capsules idle past `MEM_AUTO_PROMOTE_AGE_DAYS` (default 3) with `decay_score ≤ MEM_AUTO_PROMOTE_DECAY_THRESHOLD` (default 0.5), hourly via `MEM_AUTO_PROMOTE_INTERVAL_SECS`; `Preference`/`Workflow` excluded. Legacy `MEM_AUTO_PROMOTE_ENABLED` still honored — truthy redundant now, falsy opts out. Manual one-shot: `POST /reviews/auto_promote {dry_run:false}`), `MEM_MAX_INGEST_PER_SESSION` (per-session ingest throttle, default unbounded; set to a positive int to reject ingests beyond that count for one session_id; `0` treated as unset to avoid the footgun of "block all ingest". Counter is process-local — resets on restart. Idempotent re-ingests don't consume slots), `MEM_LAST_USED_FLUSH_SECS` (last-used worker drain cadence, **default 5**; roadmap O1 retrieval reinforcement — the **always-on** `last_used_worker` coalesces the capsule-used events `search` emits for every capsule written into a response and stamps `last_used_at` off the read path via `Store::bump_last_used_at`, one batched flush per tenant per tick. `last_used_at` anchors the decay clock: `apply_time_decay` decays a row from `last_used_at` if it was ever used, else `updated_at`, so retrieved capsules decay slower than untouched ones. The sweep advances `last_used_at` (not `updated_at`) each tick, so `updated_at` is now a pure write-freshness signal. See `src/worker/last_used_worker.rs`), `MEM_INGEST_NEARDUP_ENABLED=1` / `MEM_INGEST_NEARDUP_THRESHOLD` (roadmap O2 write-time near-duplicate review flagging, **default OFF** / threshold **0.92**; when on, the embedding worker — right after it embeds a freshly-ingested `Active` capsule — vector-searches its nearest neighbor and, if cosine ≥ threshold, flips the new capsule to `PendingConfirmation` and records a `suspected_supersede` graph edge for review (verbatim-safe: no auto-merge). Runs off the ingest path; threshold lives on `EmbeddingSettings`, invalid values silently fall back. See `src/worker/embedding_worker.rs::flag_if_near_duplicate`), `MEM_RECALL_PER_SOURCE_CAP` (roadmap O3 recall diversity cap, **default 3**, `0` disables; `pipeline/retrieve.rs::finalize` keeps at most N capsules per *source* (`session_id`, else the capsule's own id) in the ranked head and defers the overflow to the tail — a **soft** cap (nothing dropped, the token budget still reaches the tail) that stops one session's batch of near-dup capsules from monopolising the top. Read live in `per_source_cap()`, invalid → default. Opposite direction from transcript session co-occurrence), `MEM_RECALL_POOL_LIMIT` (lifecycle-pool cap, **default unset = unbounded** = the full active pool; `0`/invalid also unbounded. When set to N>0, `search_candidates` only loads the N most-recently-written non-guidance rows into the lifecycle pool — `Preference`/`Workflow` guidance is **always** included (floor-exempt). Hybrid BM25/ANN hits are unioned in by `retrieve` regardless of the cap, so the only thing bounded out is the Rust-side lexical/scope/graph *fallback* over very old non-hybrid-matched capsules. Caps the per-search row fetch as the corpus grows — a scale knob, not needed at small corpora. Read live in `LanceStore::search_candidates`/`pool_bound_clause`), `MEM_RECALL_STYLE` (UserPromptSubmit auto-recall banner style, **default `index`** since 2026-06-12 — progressive disclosure: one `source_summary` headline (≤80 chars) + `[mem_…]` id per hit, agent fetches verbatim content via `capability_capsule_get` on demand; measured ~45% smaller banners (54% on the largest) vs the legacy 240-char-body shape. Set to `snippet` to roll back to full snippets+code_refs. Directives always render full-text in both styles. COUPLING: the banner format is parsed back by `cli/feedback.rs::scan_transcript` — round-trip tests in `cli/feedback.rs` feed the renderer's verbatim output into the parser; change them together). The legacy `MEM_VECTOR_INDEX_*` and `MEM_TRANSCRIPT_VECTOR_INDEX_FLUSH_EVERY` env vars were removed in `docs/backend-coupling.md` §4.4 QW-4 — they tuned the usearch sidecar that Lance 0.27's native ANN replaced. The `MEM_DUCKDB_THREADS`, `MEM_READ_ENGINE`, and `MEM_RW_POOL_DISABLED` env vars were removed in route-B (2026-06-24) when the DuckDB read engine was deleted — reads are now lancedb-native, with no DuckDB connection, thread cap, read-engine switch, or r2d2 read pool.

## Conventions

- **Rust edition 2021** (see `Cargo.toml`). `cargo fmt` + `cargo clippy --all-targets -- -D warnings` clean is non-negotiable. snake_case for modules / files / functions.
- **Tests:** integration tests live in `tests/` at the repo root (e.g. `tests/search_api.rs`, `tests/hybrid_search.rs`, `tests/vacuum.rs`). Unit tests sit inline as `#[cfg(test)] mod tests` at the bottom of source files. **No** colocated `*_test.rs` convention in this codebase.
- **Schema:** LanceDB tables are defined inline as `Schema::new(vec![Field::new(...)])` in `src/storage/lance_store/{mod,sessions,episodes}.rs`. No external migration files. Adding/changing a table means updating the schema fn + record_batch builders + parsers in lockstep.
- **Commit scope tags:** `feat(area)`, `fix(area)`, `docs(area)`, `test(area)`, `refactor(area)`, `chore`. When closing a roadmap item: `… (closes mempalace-diff §8 #N)`.
- **Pre-commit CI check (mandatory):** Before EVERY commit, run BOTH:
  1. `cargo fmt --check`
  2. `cargo clippy --all-targets -- -D warnings`

  CI runs both gates on the full crate including `tests/` (note `--all-targets`); a clippy lint inside an integration test or bench file (e.g. `tests/bench/runner.rs`) will fail CI just like one in `src/`. Never commit if either check fails. If clippy flags a lint, fix the lint — do not silence with `#[allow(...)]` unless you have a documented reason.

## Architecture (non-obvious bits)

- **Single binary, two long-running subcommands**: `mem serve` (HTTP) and `mem mcp` (stdio MCP forwarder over `MEM_BASE_URL`), plus the one-shot CLI utilities under `src/cli/` (`mine`, `wake-up`, `feedback`). All share `Config::from_env`. MCP is *not* a separate process talking to the datasets directly — it speaks JSON-RPC and proxies to the HTTP service. Two HTTP services pointed at the same Lance dir will fight over writes; run a single `mem serve` per dataset.
- **Storage layer (`src/storage/`) — lance-native (route-B, 2026-06-24; the DuckDB read engine is gone)**: Lance datasets on disk are both the store and the read surface. Writes and reads both go through the **lancedb Rust API** (`LanceStore`, `src/storage/lance_store/`). There is no SQL engine, no `ATTACH`, no in-process DuckDB connection, no `Arc<Mutex<Connection>>`/`spawn_blocking` read serialization, and no `refresh`/`mark_dirty`/`ensure_fresh` per-connection version cache — all of that machinery was deleted with the read engine. Reads are expressed as lancedb queries: `query().only_if("<sql predicate>")` for filter scans, `query().nearest_to(vec).nprobes().refine_factor()` for vector ANN, and `fetch_*_by_ids` to hydrate full rows; ranking, aggregation, version-chain dedup, graph-BFS, and **RRF fusion** all run in Rust on top of those primitives (`pipeline/retrieve.rs::sql_rrf` re-computes RRF over the BM25 + ANN candidate-id ranks — the portable equivalent of the old fused SQL). **Read freshness:** the read connection is opened with `read_consistency_interval(0)` (Strong; `lance_store/mod.rs`, commit `67414f6`), so every read transparently sees the latest committed version via a cheap per-read manifest check — no explicit refresh call. **FTS / BM25 is a self-contained Tantivy subsystem** (`src/storage/fts.rs`), NOT a Lance index: a per-bucket (capsule + transcript) in-RAM (`Index::create_in_ram`) inverted index with a **jieba** precision-mode tokenizer for the Chinese-heavy corpus; query text is jieba-term-split into a `should`/OR `BooleanQuery` (an unspaced CJK run handed to Tantivy's `QueryParser` parses as a 0-hit phrase). The index is full-rebuilt from the source Lance table at startup and again whenever `MaintenanceStore::rebuild_query_indexes` / `ensure_query_indexes` runs (driven by the vacuum worker); at real scale a full rebuild is <1s (~6s at 10×), so there's no on-disk index, no stale-index window, and no soft-degrade machinery for BM25. (This Tantivy subsystem replaced the old lance `lance_fts(...)` table function, whose stale-index scans hit a lance-core `scanner.rs` ragged-batch bug — see the transcript-search note below.) **Decay writes** (`apply_time_decay` — hard-expiry + 2 decay passes; `bump_last_used_at`) live in `src/storage/lance_store/decay.rs` and go through the **lancedb Rust API** `table.update()` — the *same single writer* as ingest, so the old dual-writer / pruned-stale-base commit race is gone. Commit conflicts between two Rust-API writers are handled by **lance's own native retry** (`UpdateBuilder::execute_with_retry`, ~10 retries / 30 s, re-snapshotting between attempts); `LanceStore::with_lance_commit_retry` is only a thin bounded *outer* safety net (re-open the table + retry, max 3) for the extreme-contention case and the deterministic test seam. **Vector ANN indexes are NOT auto-built by Lance** — an unindexed `nearest_to` flat-scans, which made transcript search 5-11s on the large `conversation_message_embeddings` table. `ensure_vector_indexes` (`lance_store/maintenance.rs`, driven by the vacuum worker — at startup + each sweep) builds an IVF_PQ index per embedding table when absent and rebuilds when the unindexed delta grows; tables below 5k rows stay flat (sub-second anyway). Graph layer (writes + reads both in `src/storage/lance_store/graph.rs`): the `graph_edges` Lance table stores edges with `valid_from` / `valid_to` timestamps. `sync_memory_edges` writes active edges, `close_edges_for_capability_capsule` runs on supersede; reads default to `valid_to IS NULL` (active only). Point-in-time lookups go through `neighbors_within(node, max_hops, as_of)` (iterative Rust BFS, `MAX_HOPS_CAP = 3`). Caller-supplied edges write via `add_edge_direct` (preserves caller's `valid_from`); explicit fact closure via `invalidate_edge(from, predicate, to, ended_at)`. Whole-graph aggregate via `graph_stats()`. **No external sidecar** (the old usearch `<MEM_DB_PATH>.usearch` path is gone). See `docs/remove-duckdb-keep-lance.md` for the migration record.
- **Pipeline (`src/pipeline/`)** is the heart of behavior, not `service/`. Four stages: `ingest.rs` (status assignment, content_hash via sha2, graph edge extraction) → `retrieve.rs` (additive integer scoring: semantic + lexical + scope + intent + confidence + freshness − decay + graph) → `compress.rs` (token-budgeted four-section output: directives / relevant_facts / reusable_patterns / suggested_workflow) → `workflow.rs` (episode → workflow extraction).
- **Embedding pipeline is async + persistent**: writes enqueue rows in the `embedding_jobs` LanceDB table; the embedding worker consumes with retry/backoff and writes successful vectors into the Lance embedding table (queried natively via `nearest_to`). Provider trait under `src/embedding/` (embed_anything local / OpenAI / fake). Failure does not block ingest — jobs go `pending → processing → completed | failed | stale`.
- **Memory has lifecycle, not just CRUD**: `MemoryStatus = Provisional | Active | PendingConfirmation`, `supersedes_memory_id` forms version chains, `feedback_events` mutate `confidence` / `decay_score`. New code touching memories must respect status transitions — see `domain/memory.rs`.
- **CLI layer (`src/cli/`)**: home of subcommand handlers other than `serve` / `mcp` — `mine`, `wake-up`, `feedback` are the current one-shot utilities. Pattern: handler returns `i32` (process exit code); `main.rs` dispatches and `std::process::exit`s.
- **Transcript archive (parallel pipeline to memories)**: a verbatim conversation archive lives alongside `memories` with **zero shared state**. Separate table `conversation_messages` (CRUD + reads: `src/storage/lance_store/transcripts.rs`), separate queue `transcript_embedding_jobs`, separate embedding table `conversation_message_embeddings` (also Lance-native vectors, no sidecar), separate worker `src/worker/transcript_embedding_worker.rs`. HTTP entry is `src/http/transcripts.rs` (`POST /transcripts/messages`, `POST /transcripts/search`, `GET /transcripts?session_id=…&tenant=…`). `mem mine` is **dual-sink**: one transcript scan writes both extracted memories (unchanged) and every block (text / tool_use / tool_result / thinking) to the archive. MCP surface is intentionally untouched — transcript search is HTTP-only by design.
- **Lance STALE-INDEX ragged-batch on transcript ANN scans → soft-degrade (live behavior; FTS half resolved by route-B Tantivy)**: a lance index scan intermittently fails materializing a batch — `IO Error: ... all columns in a record batch must have the same length` (`scanner.rs`). **Root cause:** a **Lance bug triggered by a STALE / partially-covering index** — an index covers rows up to build time, then the transcript embedding worker keeps appending rows (the "unindexed delta"), and a scan that merges the indexed segment with the unindexed tail produces unequal-length columns. (Two earlier theories were WRONG — it is NOT (a) IVF over-partitioning and NOT (b) a DuckDB-extension-vs-lance-Rust *reader/writer version skew*: the v0.1.7 probe rerouted reads through the lancedb **Rust** API, same lance 7.0 as the writer, and the FTS scan hit the *same* `scanner.rs` ragged-batch — lance's own 7.0 reader has the bug too.) **Route-B status (2026-06-24):** FTS/BM25 no longer touches a Lance scan at all — it moved to the in-RAM **Tantivy** subsystem (`src/storage/fts.rs`), which eliminated the *FTS* half of this bug (no Lance FTS index, no stale-index window for BM25). What remains exposed is the **IVF/ANN** scan over `conversation_message_embeddings`; capsule recall is unaffected (flat-scanned, no index).
  - **Mitigation (kept, this is the live behavior):** `TranscriptService::search` (`src/service/transcript_service.rs`) **soft-degrades every lance-scan boundary** — semantic ANN, recent-browse, anchor injection, Phase-2 hydrate, per-primary Phase-5 context window — catching a lance read error with a `warn!` and degrading (sibling channel carries / empty windows / primary-only context) rather than 500ing. **Rule: any new lance-scan call on the transcript read path must go inside this soft-degrade pattern, not a bare `?`.** Net: transcript search never 500s but can silently lose a channel's ANN recall during a stale-index window until the next reindex. To *minimise* the window, keep the ANN index fresh — `POST /admin/reindex` (`MaintenanceStore::rebuild_query_indexes`) force-rebuilds (and full-rebuilds the Tantivy FTS) regardless of delta; more frequent vacuum-worker reindex shrinks it further (CPU tradeoff). Eliminating it entirely needs an upstream Lance fix to the stale-index scan.
- **Entity registry**: `entities` + `entity_aliases` tables canonicalize alias strings to stable `entity_id` (UUIDv7). `MemoryRecord.topics: Vec<String>` is the caller-supplied input; ingest pipeline (`extract_graph_edge_drafts` + `resolve_drafts_to_edges` in `service::memory_service`) routes through `EntityRegistry` so `graph_edges.to_node_id` is `"entity:<uuid>"`. Aliases are normalized (lowercase + whitespace-collapsed) at the PK; canonical_name preserves caller verbatim. Tenant-scoped, session-orthogonal.

## Design Discipline

- **Verbatim rule**: `memories.content` is the **fact source** — never rewrite, never truncate at storage. `memories.summary` is **index / hint only** — never use it as the basis for an answer or quote. Output-layer compression (`pipeline/compress.rs`) operates on `content`, never replaces it. The ingest pipeline enforces that, when a caller provides an explicit `summary` field, it must not equal `content` — agents must not copy refined/summarized text into the `content` field. When no caller summary is supplied, the server derives one from `content[:80]` for indexing purposes only.
- **Two-axis layering** (see `docs/mempalace-diff.md` §8): 📦 storage stays verbatim, 🔍 indexing / ranking / lifecycle is where structured signals live, ⚙️ infra / bug-fix is its own track. Before touching ranking, ingest, or output, name which layer you're in.

## Feedback discipline (calling agent → MCP)

When using `mem`'s MCP tools, **closing the feedback loop is part of the contract** — the lifecycle (`confidence` ↑, `decay_score` ↑, status → `Archived`) only moves when callers send signals back. A read-only consumer that never calls back means every memory's score is frozen at ingest, and ranking quality stops compounding.

The single MCP entry point is `mcp__mem__capability_capsule_feedback` (`POST /capability_capsules/feedback`): argument `feedback_kind: string`, plus an optional `note: string` persisted verbatim on the `feedback_events` row. (The old `capability_capsule_apply_feedback` alias — same backend, arg renamed `kind` — was removed; its `note` capability folded into `capability_capsule_feedback`.)

The five `feedback_kind` values and what each does (`src/domain/memory.rs::FeedbackKind`):

| `feedback_kind`         | confidence Δ | decay Δ | side effect           | when to send                                                                                       |
|-------------------------|--------------|---------|-----------------------|----------------------------------------------------------------------------------------------------|
| `useful`                | +0.10        | 0       | marks validated       | A retrieved memory **directly** unblocked / answered the current task. The strongest positive.    |
| `applies_here`          | +0.05        | 0       | —                     | Memory was relevant context but not the load-bearing fact. The mild positive.                     |
| `outdated`              | 0            | +0.20   | —                     | Memory was right at ingest but is now stale (renamed file, reverted decision, expired credential). |
| `does_not_apply_here`   | 0            | +0.10   | —                     | Correct elsewhere but doesn't fit this scope/project. Don't archive — just deprioritize.          |
| `incorrect`             | 0            | 0       | **status → Archived** | Memory contains a factual error. Permanent — same path as the admin UI's "delete".                 |

### When to fire

- Send **at most one** signal per memory per session — the strongest one. Don't spam the queue with `applies_here` for every search hit.
- Only fire on memories you actually **read and used** — search-hit-but-skimmed-and-ignored is not feedback. Silence is a valid signal too.
- `incorrect` is destructive (archives the row); reserve it for "I verified this is wrong," not "I disagree."
- The `tenant` field is required by the HTTP layer but the MCP wrapper fills it from `MEM_TENANT` automatically — leave it out of the call and the resolver picks `local`.

### What this does to ranking

Per-memory scoring (`pipeline/retrieve.rs`) sums an integer-weighted blend of semantic + lexical + scope + intent + confidence + freshness − decay + graph signals. Feedback tweaks two of those (`confidence`, `decay_score`) on the underlying record, so the next retrieval that surfaces the same memory ranks differently. There is no offline batch that "applies" feedback later; the write is immediate and visible to the next `memory_search` call.
