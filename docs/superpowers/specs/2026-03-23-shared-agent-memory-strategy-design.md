# 多 Agent 共享记忆与智能读写策略设计

日期：2026-03-23  
状态：Draft（方向已确认，待实现计划）

## 1. 目标

在保持 `mem` 作为通用记忆后端的前提下，为 **Codex 为主、兼容其他 coding agent** 的接入形态提供一层薄策略能力，使多个 agent 可以：

- 共享同一份项目记忆；
- 在任务开始和关键节点更稳定地检索相关记忆；
- 以更低污染风险写入事实、经验与偏好；
- 通过反馈逐步修正召回质量。

本设计的重点不是构建一个新的“自主大脑”，而是把现有 HTTP + MCP 能力提升为 **更一致、更可解释的共享记忆工作流**。

## 2. 推荐架构

采用三层职责分离：

| 层 | 职责 | 本期定位 |
|----|------|----------|
| `mem` 服务层 | 存储、检索、反馈、pending review、episode、graph | 保持通用后端，不引入过重策略 |
| `mem-mcp` 适配层 | 将 HTTP API 暴露为 MCP 工具 | 从“薄代理”升级为“轻策略网关” |
| agent 侧 Skill / Prompt 层 | 规定何时查、何时写、何时反馈 | 统一行为习惯，但不依赖单一宿主 |

结论：

- **共享能力** 由 `mem` + `mem-mcp` 提供；
- **智能使用 / 智能写入** 的第一版主要落在 `mem-mcp` 的高层工具设计上；
- `Codex ↔ Codex` 作为主验证路径，其他支持 MCP 的 coding agent 视作兼容客户端。

## 3. 共享模型

### 3.1 单一事实来源

- 所有 agent 指向同一个 `MEM_BASE_URL`；
- 底层由同一个长期运行的 `mem` 服务实例持有 DuckDB；
- 客户端不直接写数据库文件，只通过 HTTP 或 MCP 访问。

### 3.2 隔离维度

为避免多个 agent 互相污染，检索和写入至少应受以下上下文约束：

- `tenant`
- `project`
- 可选 `repo`
- 可选 `module`

本期建议：

- 默认只自动使用 **project 级** 记忆；
- `repo` / `personal` 级记忆需要显式开启；
- 不做跨项目自动知识迁移。

### 3.3 来源标识

每个 agent 都应提供稳定的 `caller_agent` / `source_agent`，例如：

- `codex:interactive`
- `codex:ci`
- `cursor:interactive`
- `claude-code:interactive`

这样做是为了：

- 回溯写入来源；
- 分析不同 agent 的记忆质量；
- 为未来按来源做排序或审计留钩子。

## 4. 记忆分层与默认策略

### 4.1 Scope 约定

系统应支持但不默认滥用以下层级：

- `project`
- `repo`
- `personal`

默认策略：

- 自动检索：仅 `project`
- 扩展检索：允许显式开启 `repo`
- 偏好或个人习惯：仅在显式开启 `personal` 时参与

### 4.2 写入分级

按内容性质分为三类：

| 类别 | 默认行为 | 说明 |
|------|----------|------|
| `fact` | 自动写入 | 已验证、可复用、表达稳定的事实 |
| `experience` | 建议写入 | 过程经验、排障路径、踩坑总结 |
| `preference` | 必须 review | 用户偏好、团队习惯、主观规范 |

设计原则：

- **事实少而准**，允许自动落库；
- **经验不直接当强事实**，先走候选或 episode；
- **偏好必须人工确认**，防止局部习惯被误写为全局规则。

## 5. 智能使用：检索策略

本期不追求“完全自治检索”，而采用 **轻量预检 + 关键节点追加搜索** 的半自动策略。

### 5.1 默认检索时机

1. **任务开始时** 执行一次轻量搜索  
   目标是帮助 agent 快速恢复项目背景、近期结论和直接相关约定。

2. **关键节点时** 追加搜索  
   第一版固定三类节点：
   - 实现前
   - 调试前
   - 评审 / 总结前

3. **已知目标对象时** 定向读取  
   当已有 `memory_id` 或 graph `node_id` 时，优先 `memory_get` / `graph_neighbors`，避免重复大搜索。

### 5.2 设计取舍

不采用“每次任务都大量预搜”，因为噪声高、成本高。  
不采用“完全按 agent 自主判断是否搜索”，因为容易漏检。  
因此推荐：

- 开始时做 **低预算 bootstrap 搜索**；
- 关键节点按意图做 **contextual 搜索**；
- 召回结果应以精简摘要包形式返回，而不是把原始数据完全暴露给模型。

## 6. 智能写入：内容治理

### 6.1 `fact`

适合自动写入的内容应满足：

- 已在当前任务中得到验证；
- 复用价值高；
- 不是一次性的临时状态；
- 可被稳定表述，而不是纯上下文碎片。

示例：

- 某个工具链升级后的兼容要求；
- 已验证可复用的构建命令；
- 项目中的明确接口约定。

### 6.2 `experience`

经验类内容通常是：

- 对排障过程的提炼；
- 对某类问题的排查路径总结；
- 对失败尝试的记录。

这类内容有价值，但不宜直接进入强召回主路径。第一版建议：

- 存为 `episode`，或
- 存为低权重候选记忆，待后续反馈确认价值。

### 6.3 `preference`

偏好类内容带强主观性，例如：

- 用户对默认 scope 的要求；
- 团队对 review、格式、提交流程的习惯。

这类内容必须进入 pending review，不允许自动活化。

### 6.4 拒绝写入的内容

第一版应明确过滤：

- 未验证猜测；
- 仅对单次会话有效的临时中间结论；
- 原始大段日志；
- 情绪化总结；
- 与当前项目无关的闲聊内容。

## 7. MCP 工具设计

本期不替换现有底层工具，而是在 `mem-mcp` 上补一组高层语义工具。底层 HTTP API 继续复用现有接口。

### 7.1 高层工具

| 工具 | 用途 | 默认策略 |
|------|------|----------|
| `memory_bootstrap` | 任务开始的轻量预检 | 只查 `project`，低 `token_budget` |
| `memory_search_contextual` | 实现 / 调试 / 评审节点追加搜索 | 按 `intent` 做更精确过滤 |
| `memory_commit_fact` | 写入已验证事实 | `write_mode=auto` |
| `memory_propose_experience` | 写入经验候选 | 落成候选或 episode |
| `memory_propose_preference` | 写入偏好候选 | 强制 pending review |
| `memory_apply_feedback` | 对已用记忆打反馈 | 包装现有 feedback 接口 |

### 7.2 与现有工具的关系

- 现有原始工具继续保留，兼容已有调用；
- 新接入 agent 默认优先使用高层工具；
- 文档中应明确原始工具更适合维护、调试或特殊脚本场景。

### 7.3 MCP 层新增的最小语义

MCP 高层工具内部可引入一个语义字段 `memory_kind`：

- `fact`
- `experience`
- `preference`

它不要求第一版立刻改动数据库 schema；可先映射到现有的：

- `memory_type`
- `write_mode`
- `scope`
- `episode` 路径

## 8. 错误处理与反馈闭环

### 8.1 检索失败

- 检索失败不应阻塞主任务；
- agent 看到的是“共享记忆不可用”，而不是“任务失败”；
- MCP 返回应区分：
  - 服务不可达
  - 空结果
  - 请求参数不合法

### 8.2 写入失败

- 写入失败不能被当作成功持久化；
- agent 不能在后续推理里默认该记忆已存在；
- 对 `commit` 和 `propose` 路径均如此。

### 8.3 反馈闭环

一条记忆被实际使用后，应允许回写：

- `useful`
- `outdated`
- `incorrect`

第一版只要求这些反馈成为检索排序的输入信号，不做复杂自动学习系统。

### 8.4 Review 闭环

- `preference` 必须进入 pending review；
- `experience` 若作为候选记忆写入，也应保持低权重或待确认状态；
- 这样可以避免多个 agent 共同把局部经验错误抬升成全局事实。

## 9. MVP 边界

### 9.1 本期要做

- `mem-mcp` 高层工具补齐；
- 默认 scope 策略收口为 project-first；
- 统一检索时机约定；
- 事实 / 经验 / 偏好三分流；
- feedback 路径打通；
- 兼容 Codex，保持对其他 MCP agent 的通用接入能力。

### 9.2 本期不做

- 复杂权限系统；
- 跨 agent 自动冲突解决；
- 自动知识合并；
- 跨项目自动知识迁移；
- 完整自治的记忆决策引擎。

## 10. 实现顺序建议

1. 在 `mem-mcp` 中新增 `memory_bootstrap` 与 `memory_search_contextual`。  
2. 新增 `memory_commit_fact` / `memory_propose_experience` / `memory_propose_preference`。  
3. 为高层工具补充统一字段：`tenant`、`project`、`repo`、`module`、`source_agent`。  
4. 将 `memory_apply_feedback` 接到现有 feedback API。  
5. 更新面向 Codex 的 Skill / README，规定默认搜索与写入策略。  
6. 再根据真实使用数据决定是否把部分策略下沉到 `mem` 服务。

## 11. 结论

第一版应将 `mem` 定位为 **跨 coding agent 的共享记忆底座**，并通过 `mem-mcp` 提供一层轻量但明确的策略约束：

- 让多个 Codex 会话稳定共享一套项目记忆；
- 让其他支持 MCP 的 agent 可以按同样方式接入；
- 让检索和写入行为更可解释，而不是完全依赖 prompt 运气；
- 先实现“薄而稳”的闭环，再决定哪些能力值得下沉到服务端。

这比继续堆叠裸 `memory_search` / `memory_ingest` 更适合作为多 agent 共享记忆的 MVP。
